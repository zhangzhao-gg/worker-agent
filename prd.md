# 工人 Agent 设计文档
> 新伦敦城市模拟系统 · Worker Agent Architecture

---

## 1. 整体概述

每个工人 Agent 是城市中一个独立存在的数字市民。它有自己的记忆、情绪和判断力，能感知城市状态，并在有限的信息下做出决策。

从技术上看，每个工人是一个独立的 Go 进程，配有专属的 SQLite 数据库文件，进程内部运行两个并行 goroutine。

| 维度 | 设计 |
|------|------|
| 进程 | 每个工人 = 一个独立 Go 进程 |
| 存储 | 每个工人一个专属 SQLite 文件 |
| 线程 | 进程内两个 goroutine：心跳 goroutine + LLM goroutine |
| 工作内容 | 由城市统一分配，Agent 只负责决定心跳计划 |
| 情绪系统 | 每次 LLM 运行时更新，存入 `soul` 表 |

---

## 2. 线程架构

### 2.1 心跳线程（纯执行，不推理）

心跳线程是工人的「身体」——机械、规律、不思考。它按照数据库中的心跳计划表执行，而不是按照硬编码的时间表。

| 职责 | 说明 |
|------|------|
| 扫描计划表 | 每分钟检查 `heartbeat_schedule` 表，到时间就发一次心跳 |
| 处理 news | 心跳响应中若包含 news，立即写入 `events` 表 |
| 评估紧急程度 | 调用轻量 LLM（haiku 级）判断 news 是否需要立刻唤醒主 LLM |
| 标记状态 | 心跳发送后将该条计划标记为 `done` 或 `skipped` |

> 心跳本身代表「工人正在工作」。城市服务器收到心跳后，会查询该工人的 session，获知当前工作内容。

### 2.2 LLM 线程（推理与决策）

LLM 线程是工人的「大脑」——只在关键时刻醒来，读取上下文，自主调用 Tools 完成任务。

| 触发时机 | 输入信息 | 主要任务 |
|----------|----------|----------|
| 早晨起床 | soul 表 + 昨晚摘要 | 制定今天的心跳计划，批量写入 schedule 表 |
| 白天紧急唤醒 | soul 表 + 摘要 + 触发 news | 判断是否调整当天计划 |
| 晚上睡前 | soul 表 + 今日 events | 生成日记摘要，更新情绪，安排未来唤醒计划 |
| 未来任意时刻 | soul 表 + 摘要 | LLM 可在任意一次运行中安排未来任何时间点的唤醒 |

> LLM 拥有绝对权力：可以增删改心跳计划，也可以在任意一次运行中安排未来**任何**时间点的唤醒，可以同时安排多个。  
> 唯一兜底：Go 代码层面验证时间合理性，防止 LLM 口误导致工人「消失」。

## 3. 一天的生命流程

| 时间 | 线程 | 发生的事 |
|------|------|----------|
| 早晨（默认 08:00） | LLM 线程 | 起床，读昨晚摘要，制定今天心跳计划写入数据库 |
| 白天（每 10 分钟） | 心跳线程 | 按计划发心跳，处理 news，必要时唤醒 LLM |
| 白天（偶发） | LLM 线程 | 被紧急 news 唤醒，重新评估计划 |
| 晚上（LLM 自定） | LLM 线程 | 睡前复盘，写摘要，更新情绪，安排未来唤醒时间 |

## 4. Tools 清单

LLM 以 Agent 模式运行，通过调用 Tools 与数据库和城市系统交互。Tools 列表写入 system prompt，Agent 启动时即可感知自己能做什么。

### 4.1 感知类 Tools（固定接口列表）

这些接口是工人感知城市的唯一主动手段，返回内容均为模糊描述，不含精确数字。

| Tool | 返回内容 | 现实类比 |
|------|----------|----------|
| `get_city_temperature()` | 模糊描述，如「寒冷，锅炉负荷很高」 | 感受到冷不冷 |
| `get_food_status()` | 模糊描述，如「配给紧张，可能削减」 | 食堂排队长不长 |
| `get_city_announcements()` | 执政官公告列表 | 公告栏 |
| `get_my_work_assignment()` | 今天城市分配的任务描述 | 上班打卡看任务单 |
| `get_recent_events(n)` | 最近 n 条 events 记录 | 回想最近发生了什么 |
| `get_memories(n)` | 最近 n 条私人记忆 | 回想自己之前在想什么 |

### 4.2 行动类 Tools

| Tool | 作用 |
|------|------|
| `write_heartbeat_schedule(entries)` | 批量写入今天的心跳计划 |
| `update_heartbeat_schedule(changes)` | 增删改单条或多条心跳计划 |
| `schedule_wakeup(datetime)` | 安排未来任意时间点的 LLM 唤醒，可多次调用安排多个时间点 |
| `write_narrative(text)` | 写对外叙事，同步到城市日志 |
| `write_memory(text)` | 写私人记忆，仅自己可见 |
| `update_soul(field, value)` | 直接修改 soul 表中的动态字段（mood、hope、grievance） |

## 5. SQLite 数据库结构

每个工人对应一个独立的 SQLite 文件（如 `edmund_ashby.db`），包含以下六张表。

| 表名 | 类型 | 存储内容 |
|------|------|----------|
| `soul` | 单行 | 工人的全部人格信息，静态身份 + 动态情绪，见下方详述 |
| `heartbeat_schedule` | 追加 | 每行一个心跳计划，含时间、任务提示、状态 |
| `wakeup_schedule` | 追加 | LLM 唤醒计划，含时间和触发原因，status: pending/done |
| `events` | 追加 | 心跳返回的 news 记录，含时间戳和内容 |
| `memories` | 追加 | LLM 写入的私人记忆和每日摘要 |
| `narratives` | 追加 | LLM 写入的对外叙事，会同步到城市日志 |

> `heartbeat_schedule` 的 status 字段取值：`pending` / `done` / `skipped`。  
> `wakeup_schedule` 独立成表，LLM 线程每分钟扫描此表，到时间就触发推理。

### soul 表字段详述

`soul` 是单行表（`id=1`），存储这个工人「是谁」和「现在感觉怎样」的全部信息。启动时从此表动态组装 system prompt。

**静态字段（初始化写入，一般不变）**

| 字段 | 说明 |
|------|------|
| `name` | 姓名 |
| `occupation` | 职业 |
| `background` | 背景故事 |
| `personality` | 性格描述 |
| `speech_style` | 说话方式 |
| `values` | 价值观 |
| `family` | 家人与重要关系 |

**动态字段（LLM 可随时通过 `update_soul` 直接写入新值）**

| 字段 | 说明 |
|------|------|
| `mood` | 当前心情值，100 为极好，0 为极差 |
| `hope` | 希望值，对未来的期待程度 |
| `grievance` | 不满值，对现状的愤懑程度 |

## 6. 城市事件总线与空间感知

城市中发生的每一件事都是一封带地址的邮件——便利店打折、路口撞车、工头喊人开工，全是不同主题的事件，先统一进入事件总线。

### 6.1 事件模型

| 字段 | 说明 |
|------|------|
| `type` | 事件类型 |
| `content` | 事件内容（自然语言描述） |
| `x, y` | 事件发生的二维坐标 |
| `radius` | 辐射半径（米），决定哪些工人能感知到 |

### 6.2 空间匹配

每个工人 Agent 在城市中有一个二维坐标 `(x, y)`，由城市侧统一管理。工人自身不感知坐标的存在。

事件触发时，城市侧做实时位置匹配：

```
对每个 agent:
    dist = distance(event.xy, agent.xy)
    if dist <= event.radius:
        将 news 放入该 agent 的下一次心跳响应
```

### 6.3 设计原则

| 原则 | 说明 |
|------|------|
| 城市侧管理坐标 | 工人不知道自己的精确坐标，也不需要知道 |
| 工人无感 | 对工人来说，news 还是心跳响应里带回来的，只是不再是全局广播 |
| 空间即过滤 | 远处的事件对工人不可见，感知是有限的、局部的 |
| worker-agent 无需改动 | 空间匹配完全在城市侧完成，工人的心跳/推理流程不受影响 |

> 典型配置：普通事件辐射 300 米，重大事件（如矿井塌方）辐射全城。城市侧可按事件类型配置不同的辐射半径。

## 7. 感知城市的方式

工人对城市的感知是有限的、模糊的，这符合「普通市民视角」的设定。

| 层级 | 方式 | 特点 | 典型内容 |
|------|------|------|----------|
| 被动感知 | news 推送 | 城市根据空间距离推送，工人只收到感知范围内的事件 | 有人死亡、食物断供、建筑完工 |
| 主动感知 | 感知类 Tools | 工人主动调用，模糊描述 | 温度、食物状态、执政官公告 |

### 永远不可见的信息

| 不可见内容 | 原因 |
|------------|------|
| 其他工人的 soul / 记忆 | 私人内心，不对外暴露 |
| 城市资源的精确数字 | 普通工人不知道仓库里有多少吨煤 |
| 城市的内部决策过程 | 执政官的会议不对工人开放 |

> 资源信息只返回模糊描述（如「紧张 / 正常 / 充裕」），不返回精确数字，这样 LLM 生成的叙事更自然真实。

## 8. 情绪系统

每个工人有三个情绪维度，均存储在 `soul` 表中，由 LLM 直接写入绝对值，无范围限制。

| 字段 | 含义 |
|------|------|
| `mood` | 当前心情，100 极好，0 极差 |
| `hope` | 对未来的希望程度 |
| `grievance` | 对现状的不满程度 |

| 属性 | 说明 |
|------|------|
| 更新时机 | 每次 LLM 运行时，由 LLM 自主决定是否更新 |
| 更新方式 | LLM 调用 `update_soul` 直接写入新值 |
| 当前影响 | 情绪值传入 system prompt，影响 LLM 的叙事语气和决策倾向 |
| 未来扩展 | 预留接口，后续可根据情绪触发特定行为 |

## 9. News 处理流程

| 步骤 | 执行者 | 操作 |
|------|--------|------|
| 1. 收到 news | 心跳线程 | 将 news 写入 `events` 表 |
| 2. 紧急判断 | 轻量 LLM（haiku） | 判断「需要立刻思考吗？」返回 yes/no |
| 3a. 若 yes | LLM 线程 | 立即唤醒，读取上下文，调用 Tools 决策 |
| 3b. 若 no | 无操作 | news 留在 `events` 表，等待下次 LLM 运行时处理 |

> news 大部分时间为空。城市中每一件事都是一个带坐标和辐射半径的事件，通过事件总线分发。只有处于辐射范围内的工人才会在心跳响应中收到这条 news。  
> 同一条 news，不同工人的 LLM 会根据各自的 soul 生成不同的叙事和反应，形成多样的城市生态。
